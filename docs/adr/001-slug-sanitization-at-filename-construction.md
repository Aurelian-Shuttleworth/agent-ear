# ADR 001: Sanitize slugs at filename construction, not per source

**Status**: Accepted
**Date**: 2026-06-12

## Context

Output filenames embed a slug (`{date}_{seq}_{slug}.md`). The slug has two sources: YAML frontmatter generated by the LLM during the Obsidian final pass, and manual topic input from the interactive prompt. Only the manual path was sanitized; an LLM-produced slug containing `/`, spaces, or other reserved characters produced broken or misplaced files. Reported by Lara during documentation review ("topic slug needs to be a valid file name").

Three options were considered: sanitize inside `extract_slug()` (per source), sanitize once where the filename is built, or reject invalid slugs with an error.

## Decision

A single `sanitize_slug()` helper is applied at filename construction in `save_markdown`/`save_json`. All slug sources — current and future — pass through it. The rule matches the existing manual-topic normalization (`[^a-z0-9]+` → `-`, lowercased, trimmed); an empty result falls back to `"untitled"`.

Rejection was ruled out because it would turn a cosmetic LLM quirk into a pipeline failure *after* the user has already spoken — losing their recording time over a filename.

## Consequences

- No code path can produce an invalid filename, including slug sources added later.
- LLM frontmatter is normalized silently; the `slug:` value inside the note may differ from the filename slug after sanitization.
- Anyone changing filename rules edits exactly one function.
